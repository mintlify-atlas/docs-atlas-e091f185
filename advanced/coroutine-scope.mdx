---
title: "Custom CoroutineScope"
description: "Use the @NativeCoroutineScope annotation to control the lifecycle and dispatcher of coroutines generated by KMP-NativeCoroutines."
---

By default, KMP-NativeCoroutines generates native wrappers that run coroutines on a shared default scope. For most use cases this is sufficient, but when you need to control the lifecycle or dispatcher — for example, to tie coroutines to a view model's lifetime — you can provide a custom `CoroutineScope` using the `@NativeCoroutineScope` annotation.

## The `@NativeCoroutineScope` annotation

`@NativeCoroutineScope` identifies the `CoroutineScope` property that the autogenerated native functions and properties will use. Place it on any `CoroutineScope` property inside your class:

```kotlin
import com.rickclephas.kmp.nativecoroutines.NativeCoroutineScope

class Clock {
    @NativeCoroutineScope
    internal val coroutineScope = CoroutineScope(job + Dispatchers.Default)
}
```

The annotation targets properties (`AnnotationTarget.PROPERTY`) and is retained only at source level, so it has no runtime overhead.

### Visibility requirement

The annotated property must be either `internal` or `public`. Private or protected scopes are not supported.

```kotlin
// Correct — internal visibility
@NativeCoroutineScope
internal val coroutineScope = CoroutineScope(SupervisorJob() + Dispatchers.Main)

// Correct — public visibility
@NativeCoroutineScope
public val coroutineScope = CoroutineScope(SupervisorJob() + Dispatchers.Default)
```

## Why use a custom scope

A custom `CoroutineScope` lets you:

- **Control the lifecycle.** Cancel all coroutines at a well-defined point (e.g., when a view model is cleared) by cancelling the scope's `Job`.
- **Choose the dispatcher.** Run generated wrappers on `Dispatchers.Main` or a custom dispatcher instead of `Dispatchers.Default`.
- **Share a scope.** Reuse the same scope across multiple annotated properties and functions in a class so they all cancel together.

## Default scope

When no `@NativeCoroutineScope` property is present in a class, KMP-NativeCoroutines uses a built-in default scope defined as:

```kotlin
internal val defaultCoroutineScope = CoroutineScope(SupervisorJob() + Dispatchers.Default)
```

This scope is not tied to any lifecycle, so coroutines started through it will not be cancelled automatically. For classes with a well-defined lifecycle, providing a custom scope is strongly recommended.

## Integration with KMP-ObservableViewModel

KMP-NativeCoroutines has built-in support for [KMP-ObservableViewModel](https://github.com/rickclephas/KMP-ObservableViewModel). Coroutines inside a `ViewModel` will automatically use the `CoroutineScope` from `ViewModelScope` without any additional annotation, matching the lifecycle of the view model.
